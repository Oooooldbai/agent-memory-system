# 图书馆记忆法 - 核心思想

> 建立索引 → 按需加载 → 减少token消耗

---

## 核心思想

图书馆记忆法的核心思想是：**像图书馆一样管理记忆**。

### 类比：图书馆

- **图书馆不会**：把所有书都放在阅读桌上（token消耗大）
- **图书馆会**：建立索引系统 → 按需取书 → 读完放回

### 类比：AI记忆

- **不要**：把所有记忆文件都加载到上下文（token消耗大）
- **要**：建立索引系统 → 按需加载记忆 → 加载完释放

---

## 解决的问题

### 问题1：启动慢
- 每次启动都加载所有memory文件
- Token消耗大，启动慢

**解决方案**：只加载轻量索引（<500 tokens）

### 问题2：信息丢失
- 启动时忘记项目配置
- 启动时忘记关键信息

**解决方案**：关键配置永久存储在MEMORY.md

### 问题3：无法按需加载
- 需要某个知识时，不知道在哪个文件
- 需要逐个搜索memory文件

**解决方案**：建立关键词映射表

---

## 核心原则

### 1. 索引优先
- 启动时只加载索引文件
- 不加载所有记忆文件

### 2. 按需加载
- 根据用户提及的关键词
- 动态加载对应的专题文件

### 3. 轻量存储
- 索引文件：<500 tokens
- 专题文件：按需加载

### 4. 永久存储
- 关键配置存储在MEMORY.md
- 启动时优先加载MEMORY.md

---

## 配置管理（重要！）

### 为什么需要配置管理？

AI Agent每次新会话都会"失忆"：
- 忘记GitHub仓库地址
- 忘记API Token
- 忘记项目路径
- 忘记关键配置

### 解决方案：三层配置体系

```
MEMORY.md          # 永久存储关键配置（GitHub/token/路径）
    ↓
memory/index.md    # 轻量索引（关键词映射）
    ↓
topics/*.md       # 按需加载的知识专题
```

### 关键配置必须记录

在MEMORY.md中永久存储：

```markdown
## 关键配置信息

### GitHub配置
- 仓库地址: https://github.com/username/repo
- Token: ghp_xxx（如果是私有配置）

### API配置
- API Key: tvly-xxx
- 其他密钥: xxx

### 项目路径
- 项目A: /path/to/project-a
- 项目B: /path/to/project-b

### 启动检查清单
1. 读取MEMORY.md中的关键配置
2. 读取memory/index.md中的关键词映射
3. 根据用户提及加载对应topics/*.md
```

### 启动流程

```markdown
## Session Startup

Before doing anything else:

1. Read `SOUL.md` — this is who you are
2. Read `USER.md` — this is who you're helping
3. Read `memory/index.md` — lightweight index（关键词映射）
4. **Read `MEMORY.md` — 关键配置信息（GitHub/token/路径）**
5. **检查项目配置**: 读取MEMORY.md中的"关键配置信息"部分
6. **检查任务追踪**: 读取 checkpoints/task-tracking.md（如有任务）
```

### 记忆索引结构

```markdown
# memory/index.md

## 关键词映射表

| 关键词 | 加载文件 | 说明 |
|--------|----------|------|
| "GitHub" / "token" / "配置" | MEMORY.md | 关键配置信息 |
| "金融" / "SCF" | topics/financial.md | 金融知识库 |
| "择校" / "学校" | topics/education.md | 择校数据 |
```

## Token节省效果

| 场景 | 传统方式 | 图书馆记忆法 | 节省 |
|------|---------|-------------|------|
| 启动 | 10000+ tokens | 500 tokens | 95% |
| 知识查询 | 逐个搜索 | 按需加载 | 80% |
| 配置查询 | 可能丢失 | 固定加载 | 100% |

## 适用场景

### 适合
- 知识库项目
- 长期记忆管理
- 多主题知识管理
- 需要频繁启动的场景
- 需要管理多个项目的场景

### 不适合
- 单次任务
- 短期项目
- 简单脚本

---

## 与消防队模式的关系

图书馆记忆法和消防队模式是互补的：

- **图书馆记忆法**：解决"启动问题"
- **消防队模式**：解决"执行问题"

**结合使用**：
- 图书馆记忆法存储项目配置
- 消防队模式存储任务进度
- 两者结合，形成完整的记忆+任务管理体系

---

## 实施步骤

### 第一步：创建MEMORY.md
- 永久存储关键配置（GitHub/token/路径）
- 包含所有项目的配置信息

### 第二步：创建memory/index.md
- 轻量索引（<500 tokens）
- 关键词映射表
- 每日记录列表

### 第三步：创建topics/目录
- 按主题创建知识专题
- 每个专题独立一个文件
- 按需加载，不全部加载

### 第四步：更新启动流程
- 在AGENTS.md中添加启动检查步骤
- 确保每次启动都加载MEMORY.md

### 第五步：建立检查点机制
- 配合消防队模式
- 任务进度写入checkpoints/
- 支持中断恢复

## 常见错误

### 错误1：忘记关键配置
❌ 每次都问用户GitHub地址
✅ 把地址永久存在MEMORY.md

### 错误2：启动加载所有文件
❌ 启动时加载所有memory/*.md
✅ 只加载index.md，按需加载

### 错误3：配置散落各处
❌ 配置在对话中说过就忘了
✅ 统一存在MEMORY.md
